# DX Check Strict Rule Pack Loader Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Wire serializer-native rule-pack lock verification into the DX Check engine loader so Forge/R2 cached packs are offline, hashed, provenance-bearing, and strict-mode rejectable without changing launch receipt scoring.

**Architecture:** `G:\Dx\check` owns trust decisions and rule loading. `rule-pack-lock.sr` is metadata, not a scoring rule source; cached registry packs are only loaded through lock entries. Trust failures become typed diagnostics and invalid rule-pack summaries, not fake score deductions or command execution.

**Tech Stack:** Rust 2024, `dx-check-engine`, `dx-serializer` `.sr` / `.machine`, BLAKE3, existing DX diagnostics and score model.

---

## File Map

- Modify `G:\Dx\check\src\model.rs`: add `strict_rule_packs` to `DxCheckEngineOptions`.
- Modify `G:\Dx\check\src\lib.rs`: pass strict mode into rule-pack loading.
- Modify `G:\Dx\check\src\registry.rs`: expose reusable lock entry lookup and path/trust helpers if needed.
- Modify `G:\Dx\check\src\rules\loader.rs`: read `rule-pack-lock.sr`, prefer fresh `check-rule-pack-lock.machine`, generate it only when writes are allowed, load cached packs through verified lock entries, and emit diagnostics for trust decisions.
- Modify `G:\Dx\check\tests\rule_pack_diagnostics.rs`: add red-green integration tests for strict/non-strict registry behavior.
- Optional CLI touch after engine green: `G:\Dx\cli\src\dx_check\engine_bridge.rs` only if compile requires option mapping.
